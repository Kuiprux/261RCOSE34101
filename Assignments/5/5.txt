Assignment 1-2: 세마포어를 사용한 공유 메모리 버퍼

s_wait()와 s_quit()를 사용한 위치
message_buffer_semaphore.c의 produce()와 consume() 함수에서 is_empty를 확인하고 message, account_id, buffer 상태에 접근하기 전에 s_wait()을, 접근과 상태 갱신을 마친 후에 s_quit()을 사용하였다.

해당 위치가 임계 구역인 이유
공유 메모리 버퍼에는 여러 프로세스가 동시에 접근할 수 있다. 이 assignment에서는 producer와 consumer가 별도의 프로세스에서 실행되고 각각 공유 메모리 버퍼를 읽거나 쓰려고 한다. 버퍼에는 message.sender_id, message.data, account_id, is_empty 값이 저장되어 있는데, 이 값들을 다른 프로세스의 개입 없이 한 프로세스에서 한 번에 읽거나 써야 버퍼의 값을 잘못 읽거나 내용이 섞이지 않는다. 따라서 이 위치는 임계 구역에 해당한다.

Assignment 1-1에서 발생할 수 있는 경쟁 상태
버퍼에는 message.sender_id, message.data, account_id, is_empty 값이 하나의 메시지를 나타낸다. producer와 consumer가 동기화 없이 동시에 is_empty와 메시지 필드에 접근하면, 한 프로세스가 확인한 버퍼 상태와 실제 버퍼 상태가 달라질 수 있다. 예를 들어 consumer가 메시지를 읽는 동안 producer가 새 메시지를 작성하기 시작한다면 consumer는 이전 메시지와 새로 쓰이는 메시지를 섞어 읽을 수 있다. 또한, 여러 producer가 동시에 is_empty 값을 읽어 비어 있다고 판단하고 동시에 버퍼에 값을 저장한다면 메시지가 일부만 읽히거나 여러 메시지가 섞여 저장될 수 있다. Assignment 1-1은 세마포어 동기화를 사용하지 않으므로 이와 같이 여러 프로세스가 버퍼에 동시에 접근하면 경쟁 상태가 발생할 수 있다.


Assignment 2-1: Named Pipe Client/Server

두 개의 named pipe를 사용하는 이유
이 과제에서는 client가 server에게 정수 값을 보내고, server는 받은 값의 제곱을 계산하여 다시 client에게 보낸다. 즉 통신이 양방향으로 이루어지는데, named pipe를 하나만 사용한다면 요청과 응답이 동시에 들어올 때 섞이거나 자신이 전송한 걸 자신이 읽을 수 있다. 따라서 named pipe를 두 개 사용하여 읽는 pipe와 쓰는 pipe를 별도로 둔다면 앞서 말한 문제를 간단히 해결할 수 있다.

client-to-server pipe의 역할
client-to-server pipe는 client가 server로 보내는 요청을 전달하는 통로이다. 항상 client가 값을 쓰고 server가 값을 읽는다.

server-to-client pipe의 역할
server-to-client pipe는 server가 client에게 보내는 응답을 전달하는 통로이다. 항상 server가 값을 쓰고 client가 값을 읽는다.

server.c에 추가한 코드 설명
먼저, access()를 사용해 client_to_server pipe(NP_RECEIVE)가 남아 있는지 확인하고, 남아 있다면 unlink()로 제거한 뒤 mkfifo()로 FIFO named pipe를 생성한다. server_to_client pipe(NP_SEND)에도 동일하게 실행한다. 이후 open()을 사용해 client_to_server pipe를 읽기 전용으로 열고, server_to_client pipe를 쓰기 전용으로 연다. 반복문 안에서는 client가 client_to_server pipe로 보낸 값을 read()로 읽고, 받은 값을 제곱하여 write()를 사용해 server_to_client pipe로 client에게 전달한다. 여기서 read()의 리턴 값이 0인지를 추가로 확인하는 코드가 있다. 이는 추가로 조사한 내용 때문인데, server가 read()로 blocking 중일 때 client에서 pipe를 닫으면 server의 read()는 0을 리턴하는데, 이 경우를 처리하지 않으면 서버는 직전에 받은 값으로 제곱을 계산하여 write()로 닫힌 pipe에 값을 전달하려 한다. 그러면 커널로부터 SIGPIPE 신호를 받아 프로그램이 종료되고, 이 과정이 출력에 남는다. 따라서 read()의 리턴 값이 0이라면 client가 전송을 마치고 종료하며 pipe를 닫았다는 의미이므로 break로 반복문을 빠져나가 server를 종료한다.

client.c에 추가한 코드 설명
client에서는 pipe를 생성하지 않고 바로 open()을 사용해 client_to_server pipe(NP_SEND)를 쓰기 전용으로 열고, server_to_client pipe(NP_RECEIVE)를 읽기 전용으로 연다. 여기서, 각 open()은 O_WRONLY로 열 경우 같은 파이프가 O_RDONLY로 열릴 때까지 대기하고, O_RDONLY로 열 경우 같은 파이프가 O_WRONLY로 열릴 때까지 대기하므로 서버와 open()하는 pipe 순서를 맞추지 않으면 deadlock이 생긴다. 따라서 server는 client_to_server pipe를 먼저 읽기 전용으로 열고, client는 같은 pipe를 먼저 쓰기 전용으로 열도록 순서를 맞추었다. 반복문에서는 1부터 9까지의 값을 server로 전송한다. write()로 client_to_server pipe를 통해 server에 값을 보낸 뒤에 read()로 server가 제곱을 계산하여 server_to_client pipe로 전송한 값을 읽고, 받은 값을 화면에 출력한다.
